Shipy Manifesto

Opinionated enough to erase decision fatigue. Simple enough to outlive hype. Freedom is the KPI.

Principles

  1. Ship over polish. Today beats never.
  2. One box, one repo, one brain. Start simple; earn your complexity.
  3. Boring on purpose. Python, HTMX, SQLite, systemd, Nginx.
  4. Start with explicit SQL. Add helpers/ORM only when the problem demands it.
  5. Convention over configuration. Defaults that push you to done.
  6. Zero build chain. Earn your JS; avoid tool jungles by default.
  7. The common web is first-class. HTML, forms, POST/redirect, cookies, sessions, CSRF.
  8. Measure, don’t mythologize. Logs, timings, error counts > vibes.
  9. Freedom is the KPI. If a choice reduces time/location/mind freedom, it’s wrong.

Default Stack

  • Backend: Python (dynamic, no Pydantic ceremony)
  • Views: Jinja2 + HTMX (progressive enhancement)
  • Data: SQLite (FTS5 when needed)
  • Auth: Signed-cookie sessions, @login_required, CSRF baked in
  • Ops: systemd + Nginx on a modest VPS

Everything else is optional and must be earned by real constraints.

Why One Box

  • Fewer moving parts. Fewer failure modes and decisions.
  • Time-to-URL in hours. From concept to deploy without infra theater.
  • Fits in your head. Cognitive load goes to users, not plumbing.

Non-Goals

  • Not a microservices framework.
  • Not a SPA scaffold or build-tool playground.
  • Not a “choose-everything” kitchen sink.

Design Promises

  • Stable defaults. Sensible structure, auth, CSRF, forms.
  • Copy-paste wins. Tutorials and examples that run on first try.
  • Escape hatches. You can drop to raw SQL, plain HTML, or add libs without a fight.
  • Dogfooding. Real apps run on Shipy; incidents improve the defaults.
  • Upgrades you can trust. Tagged releases, notes, and a deprecation path.
